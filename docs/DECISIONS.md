# 決定ログ

## 2026-06-27: 第0回 — MVP と土台方針

### MVP の狙い
- 選択: 「3D地形 × 3Dヘックスの合成プロトタイプ」。移動・表示の核は仕様書で見えているため、次は地形との合成の手触りを検証する。
- 理由: 体験の核が「地形分割・移動・表示が気持ちよくハマる」ことだから、合成が成立するかを最初に確かめるのが最短。
- 可読性A/B(仕様13章)は同時に少しだけ触れるが、本格検証は次イテレーション。

### 地形の表現形式
- 選択: **真の3Dボリューム**(オーバーハング・洞窟を含みうる)。
- 理由: 「3Dをヘックスで分割する」面白さを最大化でき、体験の核と一致する。
- 却下/保留した案: 高さマップ(オーバーハング不可で3Dの旨味が減る→却下)、実験用の固定形状手置き(却下はせず、ボリュームの初期サンプル形状として利用してよい)。

### 地形とセルの重なり判定
- 選択: **セル中心から距離 d 以内に地形があれば進入禁止 + 遮蔽**。
- 理由: ユーザー指定。`d` を調整軸として持て、手触りのチューニングがしやすい。
- 保留した案: 胞(菱形十二面体)の体積占有率(より厳密、後で厳密化候補)、中心の地形内外二値(粗い)。

### 技術前提
- 選択: Web 版(Three.js / React Three Fiber 系)先行。
- 理由: 足場を早く固める。展開先を見てからエンジン採用を検討する。

### 設計フェーズに持ち越す論点(テックリードへ)
- 真の3Dボリュームの内部表現(SDF / ボクセル / メッシュ)と、「中心から距離 d 以内に地形」を効率よく判定する方法。
- 地形の見せ方(ボリューム表示 vs 占有セルの菱形十二面体塗り)。

## 2026-06-27: 設計フェーズ — 地形表現の確定

### 地形の内部表現と距離d判定
- 選択: **SDF(符号付き距離関数)**。判定は `sdf(セル中心) ≤ d` の1式。
- 理由: dの実時間調整が定数1個の変更で済む / オーバーハング・洞窟の1形状を解析合成で容易に作れる / 描画も同じSDFを Marching Cubes に通すだけで、分類と表現が同一ソースから出て不整合が原理的に起きない / it-1 のセル数(数千)では再計算は active・パラメータ変更時のみで最適化不要。
- 却下/保留: ボクセル(分類と描画のソースが分離し不整合リスク)、メッシュ(距離d判定にBVH最近接が要り it-1 には重い)。
- 将来差し替え: `Terrain` インタフェース(`sdf(p)` / `mesh()`)の裏で Voxel/Mesh に置換可能な構造を DESIGN.md に確保済み。

### 採用スタック(テックリード推奨を承認)
- Vite + TypeScript + React Three Fiber(+ drei)、状態管理 zustand、パラメータUI leva。
- 全ロジックは格子座標空間。`worldPos`(S込み)は描画専用に閉じる。マーカーは InstancedMesh。

## 2026-06-29: 構想転換 — 廃墟大聖堂 / 壮大スケール / 飛行(it-4)

### 世界観
- 地形 = **廃墟化した大聖堂**(壁が一部崩れ内部露出)。崩落エントランスルームより大幅に大きい。
- ユニット = 人間サイズ ≈ 1セル。飛行できる翼人・魔女のファンタジー設定。

### スケール
- 選択: **壮大**(身長数十倍の大空間)。セル数は数万規模になりうる。
- 理由: 大聖堂の壮大さと飛行の立体機動を出すため。コスト承知(d 再計算・描画に最適化が要る可能性)。

### 移動モデル
- 選択: **全ユニット飛行**。室内外の3D空間すべてが移動可能、地形=実体occluderのみ。
- 実装含意: 現状の `reachable`(近傍 ∩ arena − occluder − 占有)が既に飛行モデル(床支持を要求しない)。**移動ロジックの変更は不要**。
- 副次効果: 床歩行前提が消え、it-3 の「瓦礫を低層 occluder にできない(過半フリーと衝突)」問題は解消。瓦礫・柱・崩落塊は素直に occluder。

### 描画戦略の転換(壮大スケール対応)
- it-1 流「全空きセルを小球で描く」は数万セルで破綻 → **廃止**。
- 代わりに: 地形 occluder(大聖堂シェル)+ active の到達セル(クリック対象)+ 脅威 + ユニット を描く。空きセル全部は出さない(必要なら active 近傍だけ薄く)。
- クリックのレイキャストは pointermove 毎でなく **click 時のみ**に。d 変更は**スロットル**して再計算頻度を抑える。

## 2026-06-30: 地形オーサリング層の導入(it-5)

### 詰まりの診断(オフラインで実描画を確認して確定)

## 2026-06-30: スケール見直し + 巨大プリセット(it-5 続き)

### 指摘(ユーザ)
- 1セル=人型1体で見るとサイズ感が合わない。既定大聖堂は身廊高 ~12セル(=人間の約12倍)で、VISION の「身長数十倍の壮大スケール」に届かず「大きめの礼拝堂」止まり。アーチ開口も人間1体がやっと通る幅で窮屈。

### 座標とアリーナの対応(確定)
- `horizRadius = hypot(a, b)`(フレーム水平2軸)、`layer L = u·√3/2`。アリーナは **水平半径 Hmax の円柱 × u∈[2Lmin/√3, 2Lmax/√3]**。身廊長手 a=±naveHalfLen をくるむため Hmax≈naveHalfLen+余白が要る。
- `recommendedArena(spec)` を新設し、地形フットプリント＋飛行余白から Hmax/Lmax を導出。store の既定はプリセットの推奨アリーナから引く(手書き定数を廃止)。

### スケール改訂
- 既定 `RUINED_CATHEDRAL` を人間スケールへ拡大: 身廊長 32 / 全幅 16 / ヴォールト頂 ~20セル(=身長の約20倍)。アーチも数セル幅で人型が悠々通る。推奨アリーナ Hmax27 / L0..21 ≈ **29k セル**。
- d はプリセットに紐付け(壁厚に応じ 0.55)。Rthreat も空間拡大に合わせ 3 へ。

### 巨大プリセット(システィーナ型)
- `aisles` フラグを `CathedralSpec` に追加。`false` で **側廊を持たない単一大空間**(背の高い窓が並ぶ一室の大ホール)を生成。
- `SISTINE_CATHEDRAL`: 身廊長 48 / 幅 12 / ヴォールト頂 ~34セル(=身長の約34倍)。天井が複数箇所抜け、壁が大きく崩れ、巨大な瓦礫が積もる。推奨アリーナ Hmax37 / L0..33 ≈ **85k セル**。
- `PRESETS`(ruined / sistine)+ `DEFAULT_PRESET` を登録。store / preview / 将来の UI から参照。
- `npm run preview:terrain [presetKey]` で任意プリセットを描画(巨大は res 上げ)。出力は `docs/qa/preview/<key>/`。

## 2026-07-01: 移動が重い回帰の修正(it-5 続き)

### 症状(ユーザ)
- 移動球クリック後に一呼吸おいて位置がジャンプ。ドラッグ回転は問題なし。スケール拡大後に顕在化。

### 原因
- `setActive`(移動)が `refresh → derive` を呼び、**毎回 `computeOccluders` で全アリーナ(29k)セルの sdf 判定を再計算**していた。occluderSet/threatSet は active 非依存(arena・terrain・d・enemies のみに依存)で、移動で変わるのは reachableSet(近傍~12セル)だけ。旧2735セルでは軽かったが10倍で秒オーダのブロックになり、`set({active})`→補間開始の直後に同期ブロックが入り、アニメ前半が飛んで「一呼吸→ジャンプ」に見えていた。
- さらに sdf 自体が重かった: breach/rubble のチャンク位置を**毎評価で rng＋三角関数**(`rotateAxisAngle`)生成していた。

### 修正
1. `setActive` は **reachableSet だけ再計算**(`set({active, reachableSet})`)。occluder/threat は据え置き(参照不変)。→ 移動コスト 約6.3s → 0.02ms。
2. cathedral.ts: breach/rubble のチャンクは seed で決定的なので **spec ごとに1度だけ前計算**し、field は前計算済み球/箱への距離のみ評価(rng/trig をホットループから排除)。屋根スタブの `tiltedBasis` も spec スコープへ。
3. **遠方セルの早期棄却**: field 冒頭で bounds 外の AABB 距離 >1.5 なら即 return(真の sdf もそれ以上で確実に空気)。アリーナは大聖堂より遥かに広い円柱なので大半の外気セルがここで抜ける。MC は bounds 内を sampling するので等値面に影響なし。
- 効果(vite-node 実測): init/d変更の occluder 判定 6284ms → 873ms(約7倍)。ブラウザ warm JIT では更に速い。occluders 数・mesh 形状は不変。

### 検証
- typecheck / 64テスト(移動で occluder/threat を再計算しない参照不変ガードを追加) / build 緑。再描画で見た目不変を確認。

## 2026-07-02: it-6 — ゲームルール層(遊べる戦略ゲームへ)

### ユーザ確定(AskUserQuestion)
- ターン構造 = **陣営交互・全ユニット(FE式)**。戦闘 = **命中率あり**。勝利条件 = **リーダー撃破**。対戦形式 = **シンプルな敵AI**。

### ルール確定(テックリード設計・数値は初版バランス)
- **移動**: ユニットごとの Move 値で BFS(12近傍・等コスト)。通行 = arena 内・非 occluder・非占有。
  - **飛行**(魔女・有翼人・ガーゴイル・死霊王): 全通行セルを移動可。
  - **歩行**(騎士・弓兵・僧侶・骸骨兵・邪術師): 各ステップが**足場セル**(下3近傍のいずれかが occluder、または層=Lmin)に限る。
  - **浮遊付与**: 僧侶・魔女(・邪術師)のスキル。射程2の歩行味方に付与し、次の自陣営ターン終了まで飛行扱い。切れた時に空中なら直下の足場へ降着(ダメージなし)。
- **ZOC**: 各ユニットは自分の12近傍(通行セル)に ZOC を張る。**敵 ZOC セルに入った時点で移動終了**(進入は可・通過は不可)。飛行にも適用 → 空中に「壁」が立つのが 3D の見せ場。
- **戦闘**: 命中% = clamp(攻撃側hit − 防御側evade + 高低差 ±10(攻撃側が上/下層) + 支援(攻撃側隣接味方×8, 最大24) − 被支援(防御側隣接味方×8, 最大24) − 遮蔽25(射撃/魔法で射線が地形をかすめる時), 5..100)。
  - ダメージ = max(1, atk − def + 高低差±1)。命中判定は seed 付き LCG(再現可能)。
  - 射線: 線分サンプルの min sdf ≤ 0.25 → **射線なし**(選択不可)/ ≤ 0.9 → **遮蔽**(−25%)。近接は射線不要。
  - **反撃**: 防御側が生存かつ射程内なら1回反撃(反撃への再反撃なし)。弓は minRange 2 で隣接反撃不可。

### 診断(ユーザの遠近観察が決め手)
- 「奥のセル面が実寸(より小さい遠近)で断面色に塗られ、プレイヤーを隠す」— 一枚平面のキャップでは物理的に不可能。実深度で茶色を塗る唯一の経路は **stencil 不可時のフォールバック(裏面パス)**。つまりユーザのタブでは hasStencil が偽で、第6〜8回で作り込んだステンシルキャップは一度も表示されていなかった(第7・8回の「奥に見える茶色」も全てフォールバック裏面の症状で辻褄が合う)。
- 原因: WebGL コンテキスト属性は canvas 生成時にしか決められず、Vite HMR はモジュールだけ差し替えるため、gl={{stencil:true}} 導入以前に作られたコンテキストがタブ内で生き続けていた。console には毎回警告が出ていたが気づかれなかった。

### 対応
- Canvas に key を付与(gl-stencil-1)。key 変更で canvas 要素ごと再マウントされ、開きっぱなしのタブでも HMR 到達時に stencil 付きの新コンテキストが必ず作られる。three r169 が stencil パラメータを getContext へ転送することはソースで確認済み。
- フォールバックへ退避した場合は console だけでなく HUD ログにも警告を出す(「⚠ 断面表示が簡易版です」)。

## 2026-07-05: rogue-1 目視フィードバック第10回 — 進行すると壁が消える

### 診断
- stencil 有効化後、探索を進める(=発見のたびにジオメトリを再構築する)と岩の表面パスが描画されなくなり、キャップと「洞窟形の暗い穴(背景色)」だけが残る。
- 原因: 同一 BufferGeometry を4メッシュへ `<primitive attach="geometry">` で共有していた。R3F の primitive attach は付け替え時に「以前の値へ戻す」帳簿処理を行い、同一オブジェクトの多重共有では差し替え時に一部メッシュが空ジオメトリへ戻ることがある。stencil 有効化で共有が2→4箇所へ増えて顕在化(計数パスは生き残るためキャップは正常、表面だけ消える=観察と一致)。

### 対応
- 共有を `geometry={geom}` プロップ渡し(単純な代入・帳簿なし)へ全面変更。R3F でジオメトリを複数メッシュに共有する正攻法。

## 2026-07-05: rogue-2 — 探索支援(手がかりの強化)

### 要求(ユーザ)
探索の手がかりが弱い。①プレイヤー層の移動可能範囲へ常時ヘックスタイル ②訪問済みの部屋の壁色マーキング+敵が居なくなった部屋は明るく(ミニマップの代替。3D形状を読めるミニマップ表現は保留) ③可視化した敵のフォーカスで距離表示 ④発見済みアイテムへ方向バブル+距離 ⑤部屋の通路入り口にもバブル ⑥バブルをクリックでファストトラベル(敵の警戒・被弾で中断)。共通機能を優先して段階実装。

- バブル(DOM/Html): アイテム=琥珀枠で名前+歩数、通路入り口=水色枠で「通路/未踏の通路」+歩数(いま居る広間のスタブのみ)。クリックで travelTo。

### タッチ(スマホ)対応の設計提案(未実装)
現状の噛み合わない点: ①ホバー依存(移動マーカーの同層ヘックス表示・敵の情報パネル)②ホイールズーム ③キーボード(TAB/M/Space)④ボタンの当たり判定サイズ。対応案:
1. **タップ2段階確定**: 1タップ目=フォーカス(ホバー相当の情報表示: 同層ヘックス・敵パネル)、2タップ目=実行(移動/攻撃)。PC のホバー+クリックと同じ内部状態(hoverMarker/hoverBeastId)に写像できるため実装差分が小さく、誤タップ防止にもなる。

## 2026-07-06: rogue-5 — 見た目と音の演出(外部アセットなし方針を維持)

### ポストエフェクト(ブルーム+ビネット)
- @react-three/postprocessing を導入。Bloom(mipmapBlur, threshold 0.55)+ Vignette + ToneMapping(ACES)。
- **stencilBuffer 付きコンポーザ**が必須: 断面キャップはステンシルに依存するため、コンポーザの FBO にもステンシルが要る(無いとキャップが全面に出る既知の壊れ方になる)。multisampling=4 で AA も維持。
- HDR パイプラインでは素通しだとトーンマッピングが外れるため、最後に ToneMapping エフェクトを足して postFx オフ時と見た目を揃える。
- 表示が重い/崩れる環境向けに HUD の ✨ ボタンでオフにできる(store.postFx)。

### BGM(WebAudio 生成アンビエント)
- 拍を持たない洞窟環境音楽4層: 低音ドローン(デチューン三角波+超低速 LFO)/ D エオリアの4和音パッド(14秒エンベロープを10秒間隔でクロスフェード)/ 水滴(ピング+フィードバックディレイ残響)/ 風(ループノイズ+揺れるバンドパス)。先読みスケジューラ(1秒毎に2秒先まで)。
- AudioContext は sfx と共有。初回操作(タイトルの「潜る」または任意の pointerdown)で開始、冪等。
- シーン連動: マップ=ローパス 850Hz+減音(俯瞰の距離感)/ 死亡=420Hz+さらに減音。ミュートは効果音と共通トグル。

### ライティング・粒子・装飾
- たいまつ: 周期の違う正弦を重ねた光量フリッカー+炎コーンの伸縮+加算グロースプライト(canvas 生成の放射グラデ、glowTexture 共有)。

- クリスタル: 訪問済み広間の下寄りセルに 2〜3個、seed+広間 id から**決定的**に配置(状態に持たない装飾。exploreRev で再評価)。発光がブルームでにじみ、部屋の色味が記憶の手がかりにもなる。

### タイトル画面と HUD 質感
- 初回のみのタイトルオーバーレイ(z=60)。「潜る」クリックが自動再生制限の解除を兼ねる。
- HUD パネルにすりガラス(backdrop-filter)+影+グラデーション、ボタンのホバーリフト、HP バーの幅トランジション。

### 見送り

## 2026-07-08: rogue-16 続き — 敵が見えないリグレッション修正+BGM ダッキング

- **敵消失の原因**: GltfBeastBody のスケール正規化が `Box3.setFromObject(clone)` を使っていた。three r151+ はスキン付きメッシュで「スキニング込みの境界」を計算するが、**描画前はボーン行列が単位行列**のため、Quaternius 系(armature に ×100 スケール)では境界が ±100〜270 に暴発 → scale≈0.001〜0.01 となり全スキン種が 1/50〜1/500 の点として描画されていた(モジュール先頭で preload するためフォールバックのプロシージャル体も出ない=完全に不可視)。KayKit の騎士は等倍作りで誤差が出ず無事だった。

- **BGM ダッキング**(ユーザ指摘: 加わるパートが小さく聞こえる): ベース層(ドローン・パッド・風・タンプーラ)を bed バスにまとめ、リズム/旋律の成長にあわせ最大4割ダッキング。加わる層は 1.3 倍・流水も 0.05→0.06 に増強。

## 2026-07-08: rogue-16 続き3 — 広間の重なり禁止

## 2026-07-11: スキル体系の改訂 — マスタリー(永続)×スロット(ラン内)

前項「次期システムの検討記録」の**スキルツリー節を置き換える**(障壁・両手持ち・層リセットの結論は有効のまま)。

### 課題
- 旧案(関門通過で SP 直取得・ラン内ツリー)は、約20ノードに対し1ランの関門通過が3〜6回で、**1ランでツリーの2割しか触れない**。スキル数と関門進行が噛み合わない。

### 決定: 二層構造(Hollow Knight のチャーム+ノッチと同型)
- **マスタリー = 横の広がり(死後も残る永続メタ)**。系統ごとに「使うことで育つ」(槍でトドメ→武技、罠発動→罠師、未覚醒討伐→隠密…)。レベルアップでその系統のノードが「選べる候補」として解禁される。稼ぎプレイ(素振り)を防ぐため、使用回数の連続加算ではなく**実績条件で1レベル**の離散式を第一候補にする。
- **スロット = 縦の力(ランごとにリセット)**。初期2、関門通過ごとに+1(上限6目安)。装着の組み替えは**関門でのみ**(「降りる前の儀式」)。
- **ドラフト**: 既存スロットの組み替えは自由、**関門で増えた新スロットは解禁済みノードから乱数3択**(乱数はシード列から)。ビルドの芯と、ランごとの変化・実験の強制を両立する。
- **ノードコスト 1〜3**: 1=数値系パッシブ / 2=行動追加系 / 3=ビルド定義級。スロット6でも全部盛りは不可能。
- バランスのノブが独立する: **強さの上限はスロット側(ラン内で完結)**。マスタリーは選択肢の幅しか増やさず、周回で難易度が壊れる方向に効かない。

### 横断ルール(バランスの骨格)
- 反撃系(受け反撃/見切り)は同時に1つ。
- 「位置をずらす」動詞は各系統に1つずつ(破砕・シールドバッシュ・投げ)。受け皿は罠師(罠の上へ動かす)に集約し、系統横断シナジーをこの1本に束ねる。
- 隠密の成功率は明かり段階に従属(広げる<普通<絞る<消灯)。灯火が enabler、隠密が payoff — 「暗いと回復が止まる」既存の芯に自動接続。
- 背水と無傷の型は同系統内の対極(捨て身 vs 完璧主義)。スロット排他。
- 一撃死(背討ち上位)は一般敵まで。門番・ボス・深層エリートは耐性(スコア上位が隠密一色になる栓)。

### 実装メモ
- 掏摸・目利きの前提: ドロップを討伐時抽選から**湧き時の決定論的事前ロール**へ変更(シード導出で再現性維持。盗んだら討伐ドロップは空=二重取り防止)。
- 背討ちの「背後」: まず覚醒状態で代用(未覚醒=背後扱い)。敵の向きの機械化は後段。
- 装備UI: 所持品パネルの装備枠を右手/左手2スロット+構え表示へ改修。

### スコアボードとの接続
- **構成併記**: 送信スキーマに死亡(または関門)時点のスロット構成とマスタリー要約を追加(スキル ID はバージョン間で安定な識別子)。top100 から「スコアを狙いやすい組み合わせ」が見え、メタ進行の目標になり、採用率がそのままバランス調整のテレメトリになる。メタ進行による不公平の懸念はこれで「学びの機能」に反転する。
- **関門スナップショット送信**: 死亡時のみだと熟達するほど長いランがボードに載らない。関門通過時に送信し、Worker はラン ID(シード+バージョン+クライアント乱数)で最深を upsert。死亡時に確定。進行中ランは「潜行中」表示。リプレイ検証は行動ログの前置き検証で成立。
- **強制終了はしない**: プレイヤーの強さは構造的に上限がある(HP24固定・スロット上限・品質上限・障壁24上書き)のに対し、敵の深度スケーリングは無制限にできる。**死は深さの必然**として訪れるので、タイマーや壁は不要。深層スケーリングがプレイヤー上限を確実に追い越すことだけ調整で保証する。自発的な終点は遺物の「脱出して確定」が担う。
- 「本日の迷宮」の固定ロードアウトは必須でなくなり、縛り開催(素手限定の日など)のバリエーション道具として保持。

## 2026-07-11: バランスシミュレータと対応順の決定

### バランスシミュレータ(ヘッドレス統計)
- **方式**: `npm run balance`(vite-node)。実ゲームと同じストアを fake timers+演出待ちゼロで数千ラン回す。ドメイン関数を叩く別実装は作らない — ストアに戦闘・討伐ロジックが残置されているため、**同じコードパスで測ることが計測の正しさ**。ゴールデンテスト機構の「1本の再現」を「シード×方策の統計」へ拡張する位置づけ。
- **ボット方策**: `(状態) → 行動` のインターフェース。貪欲型(ゴールデンの固定方針の拡張)と慎重型から始め、スキル導入後はビルド特化型(罠師・隠密・拳闘…)を足してアーキタイプ別の死亡曲線を出す。
- **指標**: 死亡深度分布(percentile)/死因別集計/種族別の遭遇・与被ダメ・キル寄与/アイテム出土・使用率/瀕死回数/層通過率/ターン数。JSON+要約テーブル。
- **本命は A/B 差分**: 変更前後で同じシード集合を回して分布を比較(層リセットや新敵で難度が意図せず動いたかを検出)。回帰ガードは緩い閾値のスモークのみ(厳密一致はゴールデンテストの仕事、こちらは統計)。
- **限界の明記**: ボットは人間の腕前の代理にならない。測れるのは下限・上限とバージョン間の相対変化。人間側の実データはローカルスコアボード(ラン履歴)が担い、2つの目を併用する。
- **副産物**: ボットの操作列=行動ログ。共有スコアボードのリプレイ検証エンジンと実行系・フォーマットを共用できる(検証器=シミュレータ)。異常なランはシード+ログがそのまま再現ケース。
