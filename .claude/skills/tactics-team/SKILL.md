# タクティクスチーム

3D タクティクスゲームを、複数の役割からなるエージェントチームとして開発するためのスキル。
仕様が未確定の前提で、**手探りで全体像と細部を詰めていく**ことを織り込んでいる。

このスキルは「進行ガイド + 必要時にサブエージェント起動」型で動く。
あなた(メインスレッド)が常時の司会・対話担当として全体を進め、対話の要らないまとまった作業だけをサブエージェントに切り出す。

技術前提: **まず Web 版(Three.js / React Three Fiber 系)で足場を固める**。展開先の環境を見て、必要になった段階で Unity / Godot などのエンジン採用を検討する。エンジンは目的ではなく手段として後から選ぶ。

---

## 起動時にまずやること

1. `docs/` 配下の既存ドキュメント(下記「成果物」)を読み、**プロジェクトが今どのフェーズにいるか**を把握する。無ければこれが第0回。
2. 直近の `docs/DECISIONS.md`(決定ログ)と `docs/ROADMAP.md` を見て、未消化のタスク・宙づりの論点を拾う。
3. ユーザーに「今どこを進めたいか」を一言で確認するか、文脈から明らかなら宣言して進む。各フェーズを律儀に頭からやり直さない — 足りていない所だけ埋める。

## 役割(チーム編成)

| 役割 | 担当(誰が演じるか) | 責務(mandate) | 主な成果物 |
|---|---|---|---|
| 要求定義ディレクター | **メインスレッド**(対話) | 「何を・なぜ作るか」を所有。ユーザーへのインタビューでビジョン・体験の核・スコープを引き出し、曖昧さを解消する | `VISION.md` / `REQUIREMENTS.md` |
| デリバリーディレクター | **メインスレッド**(調整) | 「実際に形にする」責任を所有。優先順位と各イテレーションのスコープを決め、野心と実装現実の橋渡しをし、ビジョン→実物への到達を保証する | `ROADMAP.md` |
| テックリード | メイン or サブエージェント | 設計とアーキテクチャを定め、作業を分解し、実装チームを編成して指揮する | `DESIGN.md` |
| 実装チーム | **サブエージェント**(複数可) | テックリードの分解した単位を実装する | コード + 変更概要 |
| QA | メイン or サブエージェント | ディレクター・リードと品質基準を握り、ビルドを検証し、欠陥を起票する | `docs/qa/` レポート |

役割は「帽子」であって人数ではない。小さな局面ではあなたが複数の帽子をかぶってよい。重い局面でだけサブエージェントに切り出す。

## 人間とサブエージェントの境界(最重要)

**サブエージェントはユーザーと対話できない。** 起動した瞬間からプロンプトだけを頼りに走り、途中で質問を返せない。だからこの線引きを守る:

- **対話が要る仕事はメインスレッドが担う。** 要求定義のインタビュー、スコープの合意、設計トレードオフの意思決定の確認 — これらはユーザーと往復しながらあなたが進める。
- **対話が要らないまとまった仕事をサブエージェントに渡す。** 競合・技術調査、設計案の起草、実装、QA 検証。渡すときは判断材料と完了条件を**プロンプトに全部入れる**(下記テンプレート)。
- サブエージェントが判断に迷う論点に当たったら、**勝手に決めさせず**「選択肢と推奨を添えて持ち帰る」よう指示する。決定はメインに戻してユーザーと握る。
- 並行で問題ないサブエージェントは**同一ターンでまとめて起動**して待ち時間を畳む。依存があるものだけ順に。

## ワークフロー(フェーズ)

厳密な一本道ではなく、行き来する前提。今いる地点から必要な所を進める。

1. **探索 / ビジョン**(要求定義ディレクター) — どんな体験か、なぜ面白いか、誰のためか。ユーザーへのインタビューで核を一つに絞る。→ `VISION.md`
   - 立ち上げ初期や曖昧なら、`/office-hours` を併用して発想を整理してよい。
2. **要求定義**(要求定義ディレクター) — スコープ、MVP、機能リスト、制約、非目標。何を**作らないか**まで言語化する。→ `REQUIREMENTS.md`
3. **設計**(テックリード) — Web 版前提のアーキテクチャ、データ構造、盤面/ターン/ユニットのモデル、技術選定の根拠。→ `DESIGN.md`
   - 設計を固める前に `/plan-eng-review`、UI/演出があれば `/plan-design-review` を使うと精度が上がる。
4. **編成 / 実装**(テックリード + 実装チーム) — 作業を独立した単位に分解し、実装サブエージェントを編成・起動する。
5. **QA**(QA) — 品質基準に照らして検証。Web 版なら `/qa` や `/browse` を活用。欠陥は `docs/qa/` に起票し、ディレクター・リードへ返す。
6. **反復**(デリバリーディレクター) — 結果を踏まえ次イテレーションのスコープを決め、1〜5 を回す。決定は `DECISIONS.md` に追記。

各フェーズの開始・終了で、関係する成果物を更新するのを忘れない。ドキュメントがチームの共有記憶になる。

## サブエージェントの起動方法

`Agent` ツールで起動する。役割ごとの**そのまま使えるプロンプト雛形**は `references/roles.md` にある — 起動前にそれを読み、当該フェーズの文脈で穴埋めする。

最低限、どのサブエージェントにも次を渡す:
- **役割と mandate**(あなたは○○。責務は△△)
- **読むべき成果物**(`docs/VISION.md` 等の絶対パス)
- **完了条件**(何が出来たら done か)
- **持ち帰りルール**(判断に迷う論点は決めずに、選択肢+推奨を添えて報告)
- **出力先**(更新するファイル、または報告フォーマット)

サブエージェントの最終報告はユーザーには見えない。**要点を必ずあなたが要約して伝える**。

## 成果物(チームの共有記憶)

すべて `docs/` 配下に置く。テンプレートと記入例は `references/artifacts.md` を参照。

- `docs/VISION.md` — 体験の核、ターゲット、なぜ面白いか
- `docs/REQUIREMENTS.md` — スコープ、MVP、機能、制約、非目標
- `docs/DESIGN.md` — アーキテクチャ、ゲームモデル、技術選定
- `docs/ROADMAP.md` — イテレーションとバックログ、優先順位
- `docs/DECISIONS.md` — 決定ログ(日付・論点・選択・理由)
- `docs/qa/` — QA レポート(イテレーションごと)

無ければ作る。更新は小さく頻繁に。

---

## 品質ゲート

「動く」だけでなく「ビジョン通りか」を見る。各イテレーションの締めで:

- 実装は `REQUIREMENTS.md` の対象スコープを満たすか
- QA が `VISION.md` の体験の核を損ねていないか確認したか
- 設計外の場当たり対応が紛れていないか(`DESIGN.md` と齟齬があれば設計か実装のどちらかを直す)
- 新しい決定・トレードオフは `DECISIONS.md` に残したか

ゲートで引っかかったら、デリバリーディレクターの帽子で次イテレーションに積むか、その場で直すかを判断する。
